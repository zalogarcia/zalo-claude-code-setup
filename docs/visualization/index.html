How to use this setup

Three outcomes you'll hit weekly — and the exact commands for each.

1. Install once

One command. Two minutes. Adds custom agents, skills, commands, and hooks to your local ~/.claude/. 

git clone https://github.com/zalogarcia/zalo-claude-code-setup ~/.claude-setup
cd ~/.claude-setup && ./install.sh

Full instructions in the README →

2. Personalize with CLAUDE.md

The install added ~/.claude/CLAUDE.md. Claude reads this every conversation — it's where your preferences live: your stack, your safety rules, your QA discipline, your team conventions. Without it, you'd re-explain "we use TypeScript, prefer Supabase, never force-push to main" in every new chat. With it, Claude already knows. 

code ~/.claude/CLAUDE.md

When Claude makes a mistake, tell it. The file's Learned Mistakes section grows automatically — your preferences sharpen over time without you maintaining a list.

For project-specific overrides, drop .claude/CLAUDE.md in any repo. Same format, scoped to that codebase. Project rules win over global.

What's already encoded

The starter CLAUDE.md ships with the principles that make /plan and /autopilot behave the way they do. A taste:

  • "Never push without explicit permission." Safety boundary that makes /autopilot commit freely but stop at the push gate. Surprise pushes break teammates and CI.
  • "3+ file edits → mandatory /qa-loop." Build and typecheck catch syntax; /qa-loop catches the integration bugs and wiring issues that static checks miss by design.
  • Verification Gate Function. Every claim of "done" requires a runnable command + its output in the same turn. "Should work" is how broken code gets shipped — this forces evidence.
  • Debug from live logs first. Form hypotheses after reading the actual error, not before. Saves the hours wasted on theories that contradict the runtime evidence.
  • Plans 3+ steps go to docs/PLAN.md. Long workflows survive /compact and context limits; re-read after compaction to restore state.
  • Direct/simple approach first. Agents love to propose queues, intermediaries, and abstractions for hypothetical futures. This principle keeps proposals flat and obvious.
  • Subagent discipline. A table tells Claude when to use safe-planner, qa-agent, brainstorm, live-test. The main thread orchestrates; subagents do the heavy lifting in fresh context.
  • Default stack. TypeScript + Supabase + Vercel. Stops Claude from suggesting AWS Lambda when you're a Next.js shop. Override per-project as needed.

That's a slice of what's in there. Open it. Read it. Edit it. It's yours now.

3. Ship a feature

Two terminals: one plans, one builds. You verify.

  1. Terminal 1

    Describe what you want in plain English:

    /plan add a dark mode toggle to my app

    Claude breaks the task down, writes a rollback-ready plan to .plan/plan.md, and pressure-tests it through verification gates.

  2. Terminal 1

    When /plan finishes, it prints a ready-to-paste /autopilot command. Copy it.

  3. Terminal 2

    Open a second terminal. Paste the /autopilot command. It runs the plan on its own — implements, tests, audits, loops until clean.

  4. Terminal 1

    When /autopilot finishes in Terminal 2, return to your main terminal and tell Claude: 

    autopilot finished — verify and push when it looks good

    Claude checks the work, runs tests, and waits for your push approval. Nothing leaves your machine without your okay.

  5. Terminal 1

    Before the next task, keep your conversation lean:

    /compact

4. Fix a bug

One command. Claude traces, diagnoses, fixes, validates.

  1. Terminal 1

    Describe what's broken:

    /bug users get logged out randomly after checkout

    Claude traces the full flow, identifies the root cause (not just a symptom), and stops if 3 fix attempts fail — that's a signal the architecture is wrong, not the fix.

  2. Terminal 1

    Claude applies a narrow fix at the root cause, then a QA pass confirms the bug is gone and nothing else broke.

5. Think something through

Pressure-test an idea before you commit code to it.

  1. Terminal 1

    Describe what you're wrestling with:

    /brainstorm should we move from REST to GraphQL for our public API?

    Claude applies inversion ("what if the opposite were true?"), simplification ("what would eliminate 5 steps with one insight?"), and stress-tests at 0.1× and 10× scale. You get a sharper view of the tradeoffs.

  2. Terminal 1

    Use the output to inform the next /plan. If you skip this step on hard decisions, you usually pay for it later.

6. Run QA on anything

Audit code, plans, configs — iteratively until clean. Not just inside /autopilot — standalone too.

  1. Terminal 1

    Point Claude at the thing you want audited:

    /qa-loop check the auth flow I just refactored

    Claude dispatches qa-agent to find real bugs (not theoretical ones), applies fixes for the CRITICAL/HIGH findings, then re-dispatches the audit. Loops until ## VERIFICATION PASSED or the iteration cap hits.

  2. Terminal 1

    Works on anything code-shaped — components, edge functions, migrations, config files, plans. Run it whenever "did I test it?" matters.

A few things that help

Earned wisdom from running this loop daily. Take what's useful, ignore the rest.

When redesigning, lead with images

Tell Claude what you want with examples, not just words. Screenshots from sites you admire, references from styles.refero.design, mood boards. Words alone leave too much room for interpretation.

Once you've locked a direction, ask Claude to write a design manifesto — a one-pager codifying the aesthetic. Then have it generate an HTML page with 3 design variants based on the manifesto. View them. Pick the winner. Iterate on that one for the actual component or page.

Keep context low

Long conversations get worse, not better. /compact often — between tasks, when context climbs past half, anytime focus shifts. Better to start the next task with crisp context than grind through a degraded one.

Push back. Ask questions.

When the plan doesn't feel right, say so. When you don't understand the reasoning, ask. The whole point of /plan is to surface disagreement before code gets written — not after a refactor lands.

Most of your time should be planning, not coding. The bug you didn't introduce is the cheapest one to fix.

Name your terminals

Use /rename right after opening a Claude Code terminal — /rename plan-dark-mode, /rename autopilot-dark-mode, /rename main-orchestrator. When you have 3+ sessions running in parallel, the tab title is the only thing that tells you which is which. Save yourself the 30 seconds of "wait, which one is this?" every time you switch back.

How each command works

Under the hood: the harnesses that make this setup powerful — and why most agent loops break without them.

CLAUDE.md

The context preloader. Every conversation starts here.

How it works

Claude reads ~/.claude/CLAUDE.md at the start of every conversation — before you've typed a word. It's how Claude knows your stack (TypeScript, Supabase, Vercel), your safety rules (never force-push, additive migrations only), your QA discipline (run /qa-loop after 3+ file edits), and your conventions. The setup ships a sensible default; you tailor it over time.

The file also drives the Self-Learning Protocol: when you correct Claude, it appends to the "Learned Mistakes" section automatically. No manual journal — the file gets sharper the more you use it.

Per-repo overrides live at .claude/CLAUDE.md in any project. Project rules override global rules — same format, narrower scope.

Why it's powerful

Without persistent global context, every new chat is a blank slate — you re-explain your preferences, your stack, your safety constraints, every time. CLAUDE.md closes that loop. Combined with /compact (which lets you preserve only high-quality context for the next task), it's the discipline that makes long-running multi-task days actually possible. The harness commands (/plan, /autopilot, /bug) all read this file too — so they default to your stack, your tests, your safety boundaries, automatically.

/plan

Plan with verification before any code runs.

How it works

/plan dispatches safe-planner to produce a rollback-ready plan: work units, dependencies, blast radius, and a stated verification path. The plan then goes through two parallel verification gates — brainstorm pressure-tests it (inversion, scale game, meta-pattern recognition) and outcomes-grader scores it against engineering principles. If either flags concerns, the plan gets one revision pass. The final output is a ready-to-paste /autopilot command.

Why it's powerful

Most coding agents start writing code immediately. Bugs from premature implementation are 10× more expensive than the planning step that would've prevented them. /plan forces the "think first, code second" discipline — without making you do the thinking. You get a reviewable plan you can actually push back on before any file changes.

/autopilot

Runs a plan to completion, autonomously.

How it works

Five phases: Plan → Implement → QA loop → Commit → Push gate. Each phase emits a completion marker (e.g. ## VERIFICATION PASSED) that the orchestrator routes on. Implementation work units run in parallel where safe. API failures retry with exponential backoff; a circuit breaker halts after 3 exhaustions in the same phase to avoid burning context on a flaky upstream.

Why it's powerful

Lets you context-switch to other work while it runs. You return to a finished implementation, tested and ready to push. The push gate is non-negotiable — nothing leaves your machine without your explicit approval. No babysitting required.

/bug

Trace, diagnose, fix, validate — with a hard stop.

How it works

Four-phase systematic debugging. Phase 1 understands the symptom (reproducible, observable, scoped). Phase 2 traces backward from the symptom through the call stack. Phase 3 finds the root cause — not a contributing factor — and distinguishes the two. Phase 4 stops at 3 failed fix attempts. That stop is a signal: the architecture allows the bug to exist, your fix isn't wrong.

Why it's powerful

Prevents the "tried five things, none worked" spiral that wastes hours and burns context. The 3-fix stop is a forcing function for stepping back instead of digging deeper. You either get a root-cause fix or a clear signal that the problem is structural.

/brainstorm

Pressure-test an idea before you commit code to it.

How it works

Applies four critical-thinking techniques: Inversion ("what if the opposite were true?"), Simplification cascade ("what would eliminate 5 steps with one insight?"), Scale game (designs for 0.1× and 10× scale and names the failure mode at each), and Meta-pattern recognition (if a problem appears in 3+ unrelated domains, it's a universal principle worth naming). Returns sharp tradeoffs, not vague recommendations.

Why it's powerful

Hard decisions made fast almost always cost more than soft decisions made slowly. /brainstorm gives you a structured second opinion without a human in the loop, surfacing the hidden assumptions you missed. Use it before any architectural change you can't easily undo.

/qa-loop

Audit, fix, re-audit — iteratively until clean.

How it works

A revision gate with an iteration cap. Dispatches qa-agent to find real bugs (not theoretical concerns). Fixes the CRITICAL/HIGH findings. Re-dispatches the audit. Loops until ## VERIFICATION PASSED or the iteration cap (typically 3) hits. If issue counts stop decreasing between iterations, escalates early — stalling is its own signal.

Why it's powerful

"Did I test it?" is unreliable. /qa-loop turns testing into a closed loop — fixes don't stop until the audit comes back clean. Catches integration bugs, wiring issues, and logic errors that static checks (build, lint, typecheck) miss by design. Built into /autopilot, but also runnable standalone whenever you want focused QA on a specific change.
